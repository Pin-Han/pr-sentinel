# pr-sentinel — 技術文件

> Repo 名稱：`pr-sentinel`
> 目標：用 LangGraph 建置一個自動 Code Review Agent，串接 GitHub Webhook，自動審查所有 repo 的 PR 並貼回留言

---

## 一、專案目標

**解決的問題**：PR 等人工 review 耗時，小問題（格式錯誤、潛在 bug、安全疑慮）應該在送審前就被發現。

**使用場景**：
- 任何你管理的 GitHub repo 開 PR，Agent 自動分析 diff 並在 PR 留言區貼出 review 結果
- 支援多個 repo，只需要在每個 repo 設定一次 webhook

**面試價值**：
- 展示 LangGraph 核心能力：條件分支、循環 retry、Checkpoint（HitL）
- 有真實 webhook 觸發，不是空跑 demo
- Python 實作，補足目前 TypeScript 為主的技術形象

---

## 二、技術選型

| 元件 | 選擇 | 原因 |
|------|------|------|
| Agent 框架 | LangGraph（Python）| 支援 cycle、條件 edge、Checkpoint |
| Web Server | FastAPI | 輕量、async、適合接 webhook |
| LLM | Google Gemini（gemini-2.5-flash-lite）| 低成本、支援 function calling 保證結構化輸出 |
| 部署 | Railway | 有公開 URL，persistent volume 支援 SQLite |
| GitHub 整合 | httpx.AsyncClient + Webhook | 全 async，與 FastAPI 一致 |

## 三、系統架構

```
GitHub Repo A ──┐
GitHub Repo B ──┼──→ Webhook POST /webhook/github
GitHub Repo C ──┘           │
                             ▼
                    FastAPI Webhook Receiver
                    （驗證 X-Hub-Signature）
                             │
                             ▼
                    LangGraph PR Review Agent
                    ┌────────────────────────┐
                    │  fetch_diff            │ ← GitHub API 取得 diff
                    │       ↓                │
                    │  analyze_code          │ ← LLM 分析問題
                    │       ↓                │
                    │  evaluate_quality      │ ← Evaluator 評分 0-10
                    │       ↓                │
                    │  score < 6?            │
                    │  ├─ Yes → revise       │ ← 最多 2 次 retry
                    │  └─ No ↓               │
                    │  high_risk?            │ ← LLM 偵測高風險變更
                    │  ├─ Yes → [HitL 等待] │ ← Checkpoint 暫停
                    │  └─ No ↓               │
                    │  format_review         │ ← 組合 Markdown review
                    │       ↓                │
                    │  post_review           │ ← 貼回 GitHub PR
                    └────────────────────────┘
```

## 四、LangGraph State 設計

```python
class PRReviewState(TypedDict, total=False):
    # ── 輸入（Webhook payload 填入）──
    repo: str               # "owner/repo"
    pr_number: int
    pr_title: str
    pr_description: str

    # ── Diff 資料（fetch_diff 填入）──
    diff: str
    changed_files: list[str]
    skipped_files: list[str]

    # ── 分析過程（analyze_code 填入）──
    issues: list[Issue]
    suggestions: list[Suggestion]
    summary: str

    # ── 評估與流程控制 ──
    score: int              # 0-10，evaluate_quality 填入
    revision_feedback: str  # Evaluator 的回饋（用於 retry）
    retry_count: int        # 目前 retry 次數（上限 2）
    is_high_risk: bool      # evaluate_quality 偵測
    human_approved: bool | None  # human_checkpoint 填入

    # ── 輸出 ──
    review_body: str
    review_decision: str    # "APPROVE" / "REQUEST_CHANGES" / "COMMENT"
```

## 五、各 Node 說明

### Node 1：`fetch_diff`
- 用 `httpx.AsyncClient`（封裝在 `GitHubClient`）呼叫 GitHub API 取得 PR 的 changed files
- 透過 `process_diff()` 處理：整檔包含或整檔跳過（不會截斷 hunk），自動跳過 lockfile、generated files、binary assets
- Token budget ~50K tokens（200K chars），超出的檔案記入 `skipped_files`

### Node 2：`analyze_code`
- 用 Gemini（gemini-2.5-flash-lite）分析 diff，透過 function calling（forced mode `ANY`）保證結構化輸出
- 回傳：issues（問題列表）+ suggestions（建議）+ summary（摘要）

### Node 3：`evaluate_quality`
- 用 Gemini 當 Evaluator，對 analysis 結果評分 0-10
- 評分標準：分析是否完整、有沒有遺漏明顯問題、建議是否具體可行
- `is_high_risk` 由 LLM 透過 function calling 回傳判斷（偵測 DB migration、auth 相關、infra 變更、危險操作）

### Node 4：`revise_review`（條件觸發）
- 當 `score < 6` 且 `retry_count < 2` 時觸發
- 把 Evaluator 的 feedback 注入 prompt 重新分析
- 這就是 LangGraph 的 **cycle**：revise → 回到 evaluate → 再判斷

### Node 5：`human_checkpoint`（條件觸發）
- 當 `is_high_risk = True` 且 `human_approved` 為 None 時觸發
- 用 LangGraph `interrupt()` 暫停 Agent，state 存到 SQLite
- 透過 `POST /review/resume` endpoint 恢復，Agent 從 Checkpoint 繼續

### Node 6：`format_review`
- 將結構化分析結果組合為 Markdown 格式
- 根據最高嚴重度 issue 決定 `review_decision`：high → REQUEST_CHANGES、medium → COMMENT、low → APPROVE

### Node 7：`post_review`
- 用 `GitHubClient.post_review()` 貼回 PR review

## 六、條件 Edge 設計

```python
def route_after_evaluate(state: PRReviewState) -> str:
    if state.get("score", 10) < 6 and state.get("retry_count", 0) < 2:
        return "revise_review"
    if state.get("is_high_risk", False) and not state.get("human_approved"):
        return "human_checkpoint"
    return "format_review"
```

優先級：retry（品質不足）> HitL（高風險）> pass-through

## 十、面試時怎麼介紹這個 Project

**一句話定位**：「一個用 LangGraph 建置的 GitHub PR Review Agent，使用 Gemini function calling 保證結構化輸出，有 Evaluator + cycle retry、高風險變更的 Human-in-the-Loop，以及跨 repo 的 webhook 整合。」

**展示的 LangGraph 能力**：
- **Cycle**：analyze → evaluate → revise → evaluate（score 不夠就重跑）
- **Conditional Edge**：score 夠了才往下走，高風險才觸發 HitL
- **Checkpoint / interrupt**：Agent 暫停等人工確認，確認後從斷點繼續

**設計決策亮點**：
- 用 `google-genai` SDK 直接呼叫 Gemini（非 LangChain wrapper），function calling forced mode (`ANY`) 保證 JSON 結構化
- GitHub client 用 `httpx.AsyncClient` 而非 PyGithub，保持全 async 與 FastAPI 一致
- Nodes 透過 `functools.partial` 注入外部依賴，每個 Node 只回傳更新的 state keys

**已知限制與改進方向**：
- 目前只分析 diff，不讀完整檔案上下文，可能產生 false positive
- Review 以 body-level 呈現，未來可改為 inline comment（搭配 path + line）
- Evaluator/retry 提升一致性，但準確度仍依賴 LLM 能力
